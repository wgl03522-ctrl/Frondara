你可以从正文中的任意段落发起讨论，让 AI 检查论证、提出反例、补充思路、分析证据，或生成修改建议。每场讨论都与其对应的原文、上下文和文档版本保持关联，不会因为对话变长，文档继续修改或对每段都展开讨论而轻易失去语境。

Frondara 不是放在编辑器旁边的另一个聊天框。

它将 AI 对话组织成与文档相连的讨论树，把普通批注变成可以持续互动的思考空间；再通过受 Git 启发的版本管理方式，让每次修改都可以比较、选择、应用和恢复。

在这里，文档始终是主干。

想法可以分支，上下文依然稳固，最终决定始终掌握在作者手中。

## 为什么线性 AI 聊天不适合复杂写作？

传统 AI 聊天只有一条不断向下延伸的时间线。

当你同时讨论同一篇文章中的研究假设、段落表达、引用依据、方法限制和反方意见时等不同段落的时候，不同问题很快就会混在一起。为了继续对话，你不得不反复复制原文、重新解释背景，并在越来越长的聊天记录中寻找之前形成的判断。

这会带来几个常见问题：

- 不同问题共享同一条对话历史，彼此干扰；
- AI 获得了更多上下文，却不一定获得了更准确的上下文；
- 讨论与原文逐渐分离，难以确认结论对应的是哪个版本；
- 有价值的探索被埋在聊天记录里，难以重新利用；
- AI 生成的修改容易直接覆盖正文，缺少比较和确认过程；
- 当文档持续变化时，旧讨论很快失去位置和意义。

Frondara 从文档本身组织对话：

```
Text








主文档

├── 引言第二段

│   ├── 这项主张是否需要引用？

│   └── 如何减少背景描述？

├── 方法章节

│   └── 样本选择是否存在偏差？

└── 讨论章节

    ├── 当前结论是否超出证据范围？

    └── 有哪些可能的替代解释？
```

不同段落自然产生不同讨论；同一段落也可以拥有多个彼此独立的问题。

只有当某个观点值得进一步探索时，你才需要从当前讨论中展开新的方向。分支不是额外的管理负担，而是复杂思考本来就具有的结构。

# 核心体验

> **让每个段落都能展开新的思路。**
>  *从文档任意位置发起 AI 对话，在不混淆上下文的情况下探索不同方向，再通过可控的修改流程将结果带回正文。*

## 1. 从文档中生长的非线性 AI 对话

不再把所有问题塞进同一个聊天窗口。

选择正文中的一句话、一个段落或一个章节，即可从当前位置发起独立讨论。每场讨论都知道自己从哪里开始、对应哪段文本，以及创建时使用了哪些上下文。

你可以：

- 从任意文本位置发起独立的 AI 对话；
- 围绕同一段落同时讨论多个问题；
- 从任意历史消息展开新的探索方向；
- 保持不同问题之间的上下文隔离；
- 在树状视图中查看讨论的展开过程；
- 在图状视图中观察跨段落、跨文档的关联；
- 随时返回讨论起点，比较不同思路；
- 将成熟的结论带回主文档，同时保留探索过程。

Frondara 不要求每场对话都不断分支。

大多数讨论可以保持简单、直接；只有当你确实需要比较不同假设、表达或推理路径时，分支才会出现。

> **非线性不是为了制造复杂，而是为了让复杂问题保持清晰。**

## 2. 与原文持续关联的 AI 互动批注

批注不应该只是一句留在页边的静态备注。

在 Frondara 中，每条批注都可以成为一场持续的 AI 对话。它不仅记录“说了什么”，还记录：

- 讨论针对的是哪段文本；
- 发起讨论时原文是什么；
- 使用了哪些文档上下文；
- AI 基于哪个版本提出建议；
- 当前正文发生了哪些变化。

你可以直接在批注中：

- 追问一个概念或结论；
- 检查论证是否充分；
- 要求 AI 从审稿人角度提出质疑；
- 寻找反例和替代解释；
- 判断某项主张是否需要引用；
- 区分事实、推断与作者判断；
- 生成更严谨、更简洁或更清晰的表达；
- 将讨论转化为待办、研究问题或文档笔记。

### 批注不会因为正文修改而轻易失效

当正文发生变化时，Frondara 会结合文本锚点、周围上下文和版本快照，尝试将批注重新定位到新的文本位置。

如果系统无法可靠确认新的位置，它不会悄悄把讨论附到错误的段落，而是明确提示你重新确认。

这种设计保证的不只是“批注还在”，而是：

> **批注与原文之间的关系依然可信。**

![与原文锚点持续关联的 AI 讨论](assets/editor-discussion.png)

## 3. 上下文可见、可控，而不是无限累积

更多上下文并不总意味着更好的回答。

传统聊天往往将完整历史不断发送给 AI。随着对话变长，过时的信息、无关的讨论和不同问题的假设会混合在一起，让模型更容易误解当前任务。

Frondara 将上下文作为一个可管理的结构。

每次与 AI 交互时，你可以清楚知道当前包含了哪些内容，例如：

- 当前选中的文本；
- 所在段落或章节；
- 必要的前后文；
- 当前讨论路径；
- 指定的相关批注；
- 引用的其他文档或研究笔记；
- 用户明确添加的资料。

你可以增加、移除或替换上下文，而不是让系统默认携带一切。

这使 AI 的回答更容易：

- 聚焦当前问题；
- 避免不同讨论之间相互污染；
- 保持术语和假设的一致性；
- 解释自己依据了哪些材料；
- 在文档不断变化时维持稳定语境。

**Ideas branch. Context holds.** 不只是一句品牌表达，也对应 Frondara 的核心交互原则：探索可以开放，但上下文必须可控。

![查看并控制当前 AI 讨论使用的上下文](assets/context-control.png)

## 4. AI 建议，而不是无提示地覆盖正文

AI 可以参与修改，但不应该未经确认替作者决定最终文本。

在 Frondara 中，AI 生成的改写首先作为候选建议出现。正文不会被自动覆盖。

对于每项建议，你可以：

- 查看建议与原文的具体差异；
- 继续询问修改理由；
- 要求 AI 调整其中一部分；
- 只采用某一句或某个词组；
- 完整应用整项建议；
- 保留为备选版本；
- 拒绝建议并保持正文不变。

每次应用都形成一次明确的内容变化，而不是一次不可见的替换。

这让 AI 从“自动生成器”变成一种更可靠的协作工具：

> AI 负责提出可能性，作者负责判断和整合。

![审阅 AI 修改建议并选择拒绝或应用](assets/ai-suggestion.png)

## 5.Git 模式的内容管理

复杂写作同样需要可靠的版本管理。

Frondara 借鉴 Git 的核心思想，但不会要求用户理解提交、变基或冲突解决等工程术语。

产品负责保存变化与关系，你只需要决定：

- 哪些内容仍在探索；
- 哪些建议值得保留；
- 哪些修改应该进入正文；
- 哪个版本最适合作为当前结果。

### 版本

保存文档在关键时间点的状态，记录内容如何逐步形成。

### 差异

清楚显示原文与建议之间具体改变了什么，而不只是展示一份新的完整文本。

### 候选分支

在不影响当前正文的情况下，探索不同表达、论证结构或研究假设。

### 选择性应用

不必接受整段 AI 输出。你可以只采用一个词、一句话、一个段落或一项具体修改。

### 整合

将成熟的讨论结果带回主文档，同时保留它来自哪场讨论、基于哪个版本。

### 恢复

在修改效果不理想时，回到之前的状态，而不必依赖手动复制和撤销记录。

### 比较

并排查看多个候选版本，判断哪一个更准确、更清晰，或更符合当前写作目标。

Frondara 提供的是一种 **Git-inspired writing workflow**，而不是把软件开发界面直接搬进写作工具。

## 一个更自然的 AI 写作流程

```
Text








选中正文段落

      ↓

发起 AI 互动批注

      ↓

追问、质疑或探索不同方向

      ↓

生成候选修改

      ↓

查看与原文的差异

      ↓

局部采用、完整应用或继续保留

      ↓

保留可追溯的讨论与版本记录
```

AI 不再脱离文档回答问题，而是参与到具体、连续且可恢复的写作过程之中。

# 典型使用场景

## 检查论证

选择一段结论，询问：

> 这项结论是否超出了当前证据支持的范围？

Frondara 可以围绕该段落展开独立讨论，指出隐含假设、证据缺口和可能的反例，而不会干扰其他章节的对话。

你可以紧接着对下一个段落继续进行探讨询问，而不受上一个讨论的影响：

> 这项方法是否是这个领域的公认首选？

Frondara 使用硬编码的记忆管理，可以使得LLM的注意力集中在全文与本段，而不是逻辑上关联不多但是在时间上刚刚发生的之前讨论。当然如果你想的话，你可以把任何在Frondara发生的讨论加入到上下文当中

------

## 模拟审稿意见

选择方法或讨论章节，要求 AI：

> 从严格审稿人的角度，指出最可能被质疑的三个问题。

每个问题都可以继续形成独立讨论，例如样本偏差、变量定义或替代解释。

## 比较多个表达版本

选择一个表达不够清晰的段落，让 AI 分别生成：

- 更简洁的版本；
- 更学术的版本；
- 面向普通读者的版本；
- 保留原有语气但减少重复的版本。

这些候选内容不会立即覆盖正文。你可以比较差异，并选择性采用其中的部分修改。

------

## 探索替代结构

围绕同一章节建立不同组织方案：

```
Text








当前章节

├── 按时间顺序组织

├── 按研究问题组织

└── 按证据强度组织
```

不同方案可以独立发展，直到其中一个足够成熟，再整合回主文档。

------

## 管理长期研究判断

研究结论经常会随着新证据发生变化。

Frondara 不只保存最终文本，也保留判断形成的过程：

- 当时有哪些证据；
- 提出了哪些反例；
- 为什么选择当前表述；
- 哪些问题仍然没有解决；
- 新资料出现后，哪些结论需要重新检查。

这使文档不只是最终结果，也成为一套可以继续发展的研究结构。

# Markdown 原生

你的研究内容应该属于你，而不是被锁定在某种私有数据库格式中。

Frondara 以 Markdown 作为核心文档格式：

- 文件可以直接使用其他编辑器打开；
- 文件和目录结构保持清晰可读；
- 适合 Git 管理、同步和长期归档；
- 便于迁移、搜索与自动化处理；
- 可以与现有脚本和工具链配合；
- 即使不再使用 Frondara，正文仍然完整可用。

讨论关系、文本锚点、版本索引和界面状态可以存储在独立元数据中，尽量不侵入正文内容。

核心原则是：

> **增强文档，但不接管文档。**

# Frondara 适合谁？

Frondara 面向需要长期处理复杂文档，并希望保留思考过程的人。

尤其适合：

- 论文和学位论文作者；
- 科研人员与高校教师；

- 长篇非虚构内容作者；
- 使用 Markdown 或 Obsidian 的知识工作者；
- 需要比较多个观点或文本版本的人；
- 希望控制 AI 上下文，而不是接受黑箱式生成的人；
- 希望保留 AI 协作过程，而不只是获得最终答案的人。

它不只是帮助你“写得更快”，也帮助你在复杂写作中保持结构、判断和控制。

# Frondara 不是什么？

为了保持产品边界清晰，Frondara 并不试图成为：

### 另一个通用聊天客户端

它的重点不是管理更多聊天，而是让讨论与文档结构保持关联。

### 一键生成整篇文章的工具

它更关注具体段落、具体问题和可以被作者检查的修改过程。

### 自动替代作者判断的写作代理

AI 可以建议、质疑和探索，但不会未经确认替作者决定正式内容。

### 要求用户学习 Git 的工程工具

Frondara 借鉴版本、差异和分支思想，但将复杂的底层操作转化为适合写作的交互。

### 封闭的文档平台

Markdown 文件应当保持开放、可读、可迁移，并能够继续在其他工具中使用。



# 我们相信

AI 不应该只在文档之外回答问题，也不应该未经确认替你改写正式内容。

更合理的方式是：

> **让 AI 对话从具体文本中生长，让不同思路保持独立，让上下文始终清晰，并让最终决定掌握在作者手中。**

Frondara 正在探索一种不同于线性聊天的 AI 交互方式：

- 文档是主干；
- 段落是起点；
- 批注是入口；
- 对话可以分支；
- 上下文可以控制；
- 修改可以比较；
- 结果可以选择；
- 过程可以保留。

思想可以不断生长，但不会因此失去结构。

## 常见问题

### 支持哪些 AI 模型？

Phase 1 将支持主流的 API 服务（OpenAI, Anthropic）。Phase 4 会增加：

- 本地模型（Llama, Mistral 等）
- 自定义 API endpoint
- 多模型混合使用

你始终可以使用自己的 API Key，Frondara 不会作为中间层或收集你的 AI 对话。

### 与 Obsidian 的关系是什么？

Frondara 可以独立使用，也可以作为 Obsidian 的补充工具。它们的关系是：

- **兼容：** Frondara 可以直接打开 Obsidian vault，识别 Wiki Links 和双向链接
- **互补：** Obsidian 专注知识管理和笔记，Frondara 专注文档写作和 AI 协作
- **独立：** 如果不使用 Obsidian，Frondara 仍然是完整的 Markdown 写作工具

未来可能会提供 Obsidian 插件，让两者更深度集成。

### 会支持实时协作吗？

不会像 Google Docs 那样的实时协作，但会支持：

- 基于版本的异步协作（类似 Git PR）
- 讨论的分享和评论
- 分支合并和冲突处理

我们认为研究写作更适合"独立思考 + 异步审阅"的模式，而不是实时共同编辑。

## 开发路线图

### Phase 1: 核心交互验证（当前阶段）

**目标：** 验证非线性 AI 对话和文档关联的核心体验

-  Markdown 文档编辑器
-  段落级批注创建
-  批注中的 AI 对话
-  从任意消息展开新分支
-  树状/图状讨论视图
-  AI 修改建议与差异对比
-  基础版本管理

**交付物：** 可用的原型，支持单文档的非线性 AI 讨论

------

### Phase 2: 版本管理与应用

**目标：** 完善 Git 式的修改管理和分支操作

-  文档快照和版本历史
-  分支创建和切换
-  选择性应用 AI 建议（按句、按段）
-  差异可视化增强
-  讨论结果与正文的合并
-  历史版本回溯

**交付物：** 完整的版本管理系统，支持复杂的修改探索和应用流程

------

### Phase 3: 知识库集成

**目标：** 与现有 Markdown 工作流深度整合

-  Obsidian vault 兼容
-  Wiki Links 和反向链接支持
-  标签和 Frontmatter 解析
-  跨文档的讨论引用
-  笔记库检索作为 AI 上下文
-  批注导出为独立笔记

**交付物：** 可作为 Obsidian 补充工具或独立工作空间使用

------

### Phase 4: 扩展生态

**目标：** 开放扩展接口，支持自定义和专业工作流

-  扩展 API 和开发文档
-  多模型支持（OpenAI, Anthropic, 本地模型）
-  自定义提示词模板
-  Zotero 文献集成
-  LaTeX 导出支持
-  社区扩展市场

**交付物：** 可扩展的平台，支持研究者和开发者自定义工作流

------

### Phase 5: 协作与同步（长期规划）

**目标：** 支持团队协作和跨设备工作

-  分支合并和冲突处理
-  讨论的分享和协作
-  可选的云同步（端到端加密）
-  团队权限管理
-  协作批注和审阅流程

**交付物：** 适合团队的协作研究和写作平台
